\&#x20;  - governance;

\\## 10. Plan-folder lifecycle requirements







The generated Claude prompt must enforce plan-folder lifecycle rules.







Claude must verify the state of:







```text



plans/proposed/



plans/active/



plans/completed/



```







before any execution slice, implementation prompt, governance transition, or plan-status claim.







\\### Required plan-folder meaning







Use the following rules unless the project’s own rules define something stricter:







1\\. `plans/proposed/`







\&#x20;  - For drafts, RFCs, refinements, candidate plans, not-yet-executed plans, and future work.



\&#x20;  - A proposed plan is not automatically executable.



\&#x20;  - Proposed status must not be presented as approved, active, or complete.







2\\. `plans/active/`







\&#x20;  - For the plan currently being treated/executed.



\&#x20;  - Claude must identify what file is active before executing a plan.



\&#x20;  - If `plans/active/` is empty, Claude must not silently invent active status.



\&#x20;  - If the task is still permitted despite no active plan, Claude must explain the governing rule or prior approval that permits it.







3\\. `plans/completed/`







\&#x20;  - Only for plan-mode files whose implementation is 100% complete.



\&#x20;  - Do not move a plan to completed because a slice, audit, proof, or partial implementation succeeded.



\&#x20;  - Completion requires full implementation, validation, and closure according to project rules.







\\### Additional plan-file rules







Claude must also enforce:







\\- Do not execute directly from `proposed/` unless the project rules or user approval explicitly allow that.



\\- Do not place partial slice evidence in `completed/`.



\\- Do not mutate active/completed/proposed plans unless the prompt explicitly allows that.



\\- Use reconciliation/provenance notes for slice evidence when that is the established project pattern.



\\- Report any mismatch between plan-folder state and claimed status.



\\- Classify plan-folder issues as BLOCKER/WARNING/DEFERRED/RESOLVED.



\\- Ask/report missing details when the project’s plan lifecycle rules are unclear.

\\## 11. tree/index synchronization requirements







The generated Claude prompt must require Claude to evaluate whether a `tree/` or equivalent index should be created or refreshed.

Claude must use a hybrid approach:







\\- tree/index for fast orientation;



\\- live grep/direct reads/git/source files for exact verification.







\\### tree/index creation or refresh triggers







Claude must create or refresh tree/index files when appropriate, especially after:







\\- governance audit drift/state change;



\\- new plan/provenance commits;



\\- phase activation/closure/acceptance;



\\- engine/skill/pack/verifier changes;



\\- new, moved, or deleted execution-path files;



\\- pack schema/artifact/fixture/dependency changes;



\\- cross-repo anchor changes;



\\- rulebook/architecture/taxonomy/README/CLAUDE/reconciliation changes;



\\- performance-budget or quality-gate changes;



\\- repeated multi-file scans;



\\- execution steps where tree/index is the best lookup layer.







Claude must not guess if tree/index status is unclear. It must report uncertainty.







\\### tree/index quality requirements







If tree/index is refreshed, Claude must ensure:







\\- markdown-only unless project rules allow otherwise;



\\- clear disclaimer that index is not source of truth;



\\- current anchors/base commit recorded according to project convention;



\\- dependency graph updated when dependencies change;



\\- plans/provenance/skills/engine/packs summaries are synchronized;



\\- no real brand/client/project content is copied if a firewall rule forbids it;



\\- no stale labels remain in load-bearing sections.

\\## 12. R10 / contamination firewall







When the project has a rule that forbids reusable code/skills/fixtures/docs from containing real brand, client, project, template, or design-example strings, the Claude prompt must enforce it strictly.







Claude must:







\\- keep reusable fixtures synthetic;



\\- keep real extraction/proof output in scratchpad only unless explicitly authorized otherwise;



\\- avoid committing real content, brand strings, route slugs, client names, contact data, or raw clone paths;



\\- record only abstract aggregates, counts, hashes, sizes, artifact names, and pass/fail results when required;



\\- run contamination greps over new/modified code, fixtures, docs, and tree/index files;



\\- distinguish allowed platform/runtime infrastructure strings from forbidden brand/project strings only when justified by project rules.







If contamination is found, Claude must STOP unless the task is explicitly a cleanup task authorized to remove it.

\\## 13. Read-only, write, backup, staging, and commit rules







Generated Claude prompts must clearly define the execution mode.







\\### Read-only audit prompts







For read-only audits:







\\- no edits;



\\- no staging;



\\- no commits;



\\- no deletes;



\\- no reset/clean;



\\- no dependency installs;



\\- no network/render/server unless explicitly required;



\\- no `.env` reads;



\\- only safe tests that are proven to write to auto-cleaned temp paths.







Claude must snapshot relevant state before and after safe tests when useful.







\\### Implementation prompts







For implementation slices:







\\- read-only STOP gate first;



\\- architecture governance audit before any write when required;



\\- backup before editing non-git or hard-to-rollback areas;



\\- write only allowed files;



\\- run tests;



\\- write provenance/tree docs only after implementation passes;



\\- stage exactly the intended docs;



\\- verify staged set;



\\- commit only allowed files;



\\- run post-commit verification;



\\- run after-governance audit when required.







\\### Backup requirements







If editing a non-git skill/tool/module, Claude must create a timestamped backup before editing.







Backup manifest should include:







\\- timestamp;



\\- base commit/HEAD;



\\- engine/tool commit if applicable;



\\- frozen checksum if applicable;



\\- files backed up;



\\- SHA-256 hashes;



\\- byte sizes;



\\- planned new files;



\\- envSecretsRead:false;



\\- governance-audit result when applicable;



\\- rollback steps.







If backup cannot be made, Claude must STOP.

Claude prompts must include performance-budget constraints when the task could touch large repositories, clones, media, generated outputs, or runtime bundles.

\\- report skipped heavy checks and why they were safe to skip.







If the task cannot be performed safely within the performance budget, Claude must STOP and report what bounded probe or authorization is needed.
